## 1) الهدف

إعداد خطة عملية لإنشاء شاشة **دفتر الأستاذ العام** داخل موديول المحاسبة بشكل مهني قابل للتدقيق، مع:

1. رصيد افتتاحي + حركة + رصيد ختامي.
2. فلترة دقيقة (تاريخ/فرع/حساب/مرجع).
3. صلاحيات صحيحة حسب الدور.
4. أداء مناسب لبيانات كبيرة.

   - `pos/src/pages/JournalEntries.jsx`.

   - `pos/src/App.jsx` (لا يوجد مسار `/general-ledger`).

## 3) متطلبات دفتر الأستاذ المستهدف

## 3.1 متطلبات محاسبية

1. رصيد افتتاحي دقيق قبل `from_date`.
2. مجموع مدين/دائن للفترة.
3. رصيد جاري لكل حركة.
4. رصيد ختامي للفترة.
5. عرض مصدر الحركة (Order/Refund/Purchase/Manual).

## 3.2 متطلبات تشغيلية

1. فلترة حسب:
   - الحساب
   - من تاريخ / إلى تاريخ
   - الفرع
   - نوع المصدر/رقم المرجع
   - مركز تكلفة (اختياري).
2. Pagination server-side.
3. تصدير CSV/XLSX بنفس الفلاتر.
4. Drill-down لفتح تفاصيل القيد اليومي.

## 3.3 متطلبات رقابية

1. منع عرض قيود Draft (posted فقط).
2. إلزام عزل الفروع للـ manager.
3. اتساق صلاحيات `accountant` بين backend وfrontend.

## 4) خطة التنفيذ المقترحة (بدون تنفيذ الآن)

## المرحلة 0 - اعتماد السياسة (0.5 يوم)

1. اعتماد تعريف الرصيد الافتتاحي.
2. اعتماد مستوى الفلاتر المطلوبة (Company/Branch/Cost Center).
3. اعتماد سياسة عرض الحسابات التجميعية (Header):
   - منع مباشر أو السماح مع `include_children`.

**مخرج المرحلة:** وثيقة متطلبات مختصرة مع توقيع المالية.

## المرحلة 1 - تصميم API دفتر الأستاذ (1 يوم)

1. توسيع contract للـ endpoint الحالي ليعيد:
   - `opening_balance`
   - `period_debit`
   - `period_credit`
   - `closing_balance`
   - `entries_count`
   - `pagination`.
2. إضافة فلاتر:
   - `from_date`, `to_date`, `branch_id`, `company_id`, `cost_center_id`, `source_type`, `source_id`.
3. الحفاظ على التوافق العكسي مع المسار الحالي.

**مخرج المرحلة:** API spec مع أمثلة request/response.

## المرحلة 2 - تصميم واجهة General Ledger (1.5 يوم)

1. صفحة جديدة `GeneralLedger.jsx` داخل الموديول المحاسبي.
2. شريط فلاتر علوي.
3. بطاقات ملخص:
   - افتتاحي، إجمالي مدين، إجمالي دائن، ختامي.
4. جدول الحركات:
   - التاريخ، رقم القيد، البيان، المصدر، مدين، دائن، الرصيد الجاري.
5. Pagination + Export + Drill-down.

**مخرج المرحلة:** UI/UX spec + mock flow.

## المرحلة 3 - الربط بالملاحة والصلاحيات (0.5 يوم)

1. إضافة route: `/general-ledger`.
2. إضافة عنصر القائمة تحت المحاسبة.
3. توحيد صلاحيات `accountant` في الفرونت.

**مخرج المرحلة:** صلاحيات معتمدة ومصفوفة وصول واضحة.

## المرحلة 4 - الأداء والجودة (0.5 يوم)

1. مراجعة الاستعلامات للبيانات الكبيرة.
2. التأكد من وجود الفهارس المناسبة (تاريخ القيد، الحساب، الفرع، الشركة).
3. تحديد حد افتراضي للصفحة (مثل 50/100).

**مخرج المرحلة:** baseline أداء قبل UAT.

## المرحلة 5 - UAT محاسبي واعتماد (1 يوم)

1. اختبار التطابق:
   - Opening + Movements = Closing.
2. اختبار عزل الفروع.
3. اختبار تطابق رصيد حساب AP مع كشف المورد GL.
4. اختبار التصدير ومراجعة مراجع القيود.

**مخرج المرحلة:** نموذج اعتماد Go/No-Go من المالية.

## 6) المخاطر وخطة التخفيف

| الخطر | الأثر | التخفيف |
|---|---|---|
| رصيد افتتاحي غير صحيح | تضليل قرارات مالية | اختبار مقارنة مع ميزان المراجعة |
| بطء الصفحة مع بيانات كبيرة | تجربة تشغيل ضعيفة | Pagination + فهارس + limits |
| تضارب صلاحيات | وصول غير مطابق للسياسة | مصفوفة صلاحيات موحدة backend/frontend |
| كسر تكامل API الحالي | تأثير على شاشات قائمة | توافق عكسي + اختبار انحدار |

## 7) معايير القبول (Acceptance Criteria)

1. كل استعلام يعرض `opening_balance`, `period_debit`, `period_credit`, `closing_balance`.
2. الرصيد الجاري صحيح لكل سطر.
3. لا توجد قيود Draft في النتائج.
4. manager يرى بيانات فرعه فقط.
5. accountant يمكنه الوصول حسب السياسة المعتمدة.
6. الصفحة تدعم Pagination وتعمل بثبات على بيانات فعلية.
7. التصدير يطابق نتائج الشاشة بنفس الفلاتر.

## 8) القرارات المطلوبة من الإدارة قبل التنفيذ

1. هل دفتر الأستاذ يسمح باختيار حساب Header مع تجميع الأبناء أم Leaf فقط؟

3. هل cost center filter إلزامي في النسخة الأولى؟
