## 1. نظرة عامة

### 1.1 الفلسفة المحاسبية

يعتمد النظام على مبدأ **القيد المزدوج (Double-Entry Bookkeeping)** كأساس لا يُحيد عنه. كل عملية مالية — من بيع بقيمة ريال واحد إلى تحويل مخزون بين الفروع — تُسجل كقيد يومية متوازن حيث:

```
مجموع المدين (Debits) = مجموع الدائن (Credits)
```

هذا التوازن يُفرض برمجياً في الطبقة الخدمية (`AccountingService.createJournalEntry`) وليس على مستوى قاعدة البيانات، نظراً لأن SQLite لا يدعم قيود `CHECK` عبر صفوف متعددة.

### 1.2 البنية التقنية

```
┌─────────────────────────────────────────────────────┐
│                   الطبقة التطبيقية                   │
│    Routes: orders, refunds, purchases, expenses     │
└──────────────────────┬──────────────────────────────┘
                       │ يستدعي
                       ▼
┌─────────────────────────────────────────────────────┐
│              AccountingHooks (الخطافات)               │
│  onOrderCompleted, onRefundApproved, onShiftClosed  │
│     (غير حاجب — لا تُعطل العمليات التجارية)         │
└──────────────────────┬──────────────────────────────┘
                       │ يُوجه إلى
                       ▼
┌─────────────────────────────────────────────────────┐
│           AccountingService (المحرك الأساسي)          │
│  createJournalEntry, recordSale, recordCOGS, ...    │
│       ← المرجعية الوحيدة لكل عملية محاسبية →       │
└──────────────────────┬──────────────────────────────┘
                       │ يكتب في
                       ▼
┌─────────────────────────────────────────────────────┐
│              قاعدة البيانات (SQLite)                  │
│   gl_journal_entries ←→ gl_journal_lines ←→ accounts│
└─────────────────────────────────────────────────────┘
```

### 2.3 ملاحظات فنية على الشجرة

- **حساب 4002 (الخصومات)**: يمتاز بطبيعة مدينة رغم تصنيفه ضمن الإيرادات — وهو تصميم صحيح لأنه حساب مقابل (Contra-Revenue) يُخفض صافي المبيعات.
- **حساب 1005 (عهدة صندوق)**: حساب وسيط حرج. يمثل المبلغ المُسلّم للكاشير عند بداية الوردية. رصيده الطبيعي يجب أن يكون **صفراً** عند إقفال جميع الورديات.
- **حساب 1105 (وسيط الفروع)**: حساب تقاص ذاتي — رصيده الصافي لجميع الفروع يجب أن يكون **صفراً** دائماً.
- **الحسابات الرئيسية (Headers)**: محمية برمجياً ولا يمكن الترحيل إليها مباشرة.

## 3. محرك القيد المزدوج (Double-Entry Engine)

### 3.1 الدالة الأساسية: `createJournalEntry`

هذه هي القلب النابض للنظام المحاسبي. جميع الدوال الأخرى تُفوّض إليها. تتضمن:

#### ضوابط التحقق (Validation Rules):

| # | الضابط | التفصيل |
|---|--------|---------|
| 1 | حد أدنى للأسطر | ≥ 2 أسطر (مدين ودائن) |
| 2 | عدم السالب | المبالغ يجب أن تكون ≥ 0 |
| 3 | حصرية الجانب | كل سطر إما مدين أو دائن (ليس الاثنين معاً) |
| 4 | عدم الصفر | كل سطر يجب أن يحمل مبلغاً > 0 |
| 5 | التوازن | مجموع المدين = مجموع الدائن |
| 6 | الفترة المفتوحة | التحقق من أن الفترة المحاسبية ليست مقفلة |
| 7 | صلاحية الحساب | التحقق من وجود الحساب وعدم كونه رئيسياً |

#### آلية إعادة المحاولة (FIX-07):

```javascript
// عند تعارض رقم القيد (race condition)، يُعاد التوليد تلقائياً
while (retryCount < 3) {
    try {
        const entryNumber = await this._generateEntryNumber(date, transaction)
        journalEntry = await JournalEntry.create({ ... })
        break; // نجاح
    } catch (err) {
        if (err.name === 'SequelizeUniqueConstraintError') {
            retryCount++
            await delay(random * 100ms)
            continue
        }
        throw err
    }
}
```

### 3.2 ترقيم القيود

```
JE-{السنة}-{رقم تسلسلي من 6 خانات}
مثال: JE-2026-000047
```

### 3.3 مبدأ عدم القابلية للتعديل (Immutability)

القيود المرحّلة (`posted`) **لا تُعدل أبداً**. التصحيح يتم عبر:
- **قيد عكسي** (`reverseJournalEntry`): ينشئ قيداً جديداً يعكس كل الأرقام.
- يُربط القيد العكسي بالأصلي عبر حقلي `notes` و`source_id`.

## 5. التقارير المالية

### 5.1 ميزان المراجعة (Trial Balance)

- يُجمّع أرصدة جميع الحسابات من القيود المرحّلة.
- يدعم الفلترة بالفترة والفرع.
- يتحقق تلقائياً من التوازن: `totalDebits === totalCredits`.
- مُستخدم كأساس لجميع التقارير الأخرى (P&L, BS).

### تفصيل الإصلاحات الحرجة

#### FIX-04: فصل ضريبة المدخلات
**قبل:** كان المخزون يُسجل بسعر شامل الضريبة ← تضخم الأصول + تكلفة مبيعات مبالغ فيها.  
**بعد:** المخزون بالسعر الصافي + ضريبة المدخلات في حساب منفصل (1300) ← إمكانية مقاصة الضريبة.

#### FIX-06: GL كمصدر حقيقي لأرصدة الموردين
**قبل:** `Supplier.current_balance` يُحدّث يدوياً عند كل عملية ← ينحرف مع الوقت.  
**بعد:** الرصيد يُحسب من GL مباشرة عبر `getSupplierGLBalance()` ← `current_balance` أصبح مجرد كاش.

#### FIX-09: تكلفة مرتجعات بالسعر الحقيقي
**قبل:** التكلفة تُقدر بنسبة من سعر البيع ← غير دقيق (خصوصاً مع خصومات وأصناف مختلفة الهامش).  
**بعد:** التكلفة تُجمع من `StockMovement` الفعلية المرتبطة بالمرتجع ← دقة 100%.

#### FIX-11: محاسبة التحويلات بين الفروع
**قبل:** المخزون يتحرك بين الفروع بدون أثر محاسبي ← أرصدة المخزون في GL لا تعكس الواقع.  
**بعد:** قيدان يومية لكل تحويل (واحد لكل فرع) عبر حساب تقاص ← توازن تام.

#### FIX-20 + FIX-21: إغلاق الورديات القسري والتلقائي
**قبل:** `forceCloseShift` (إغلاق المدير) و`cleanupOrphanedShifts` (إغلاق تلقائي) يُغلقان الوردية بدون استدعاء `AccountingHooks.onShiftClosed()` ← لا يتم تسجيل فروقات الصندوق.  
**بعد:** كلتا الدالتين تستدعيان `onShiftClosed` بعد التحديث ← كل إغلاق وردية مغطى محاسبياً.

#### FIX-22: خطر مسار PUT /status=completed
**المشكلة:** مسار `PUT /:id/status` يمكنه تحويل طلب إلى `completed` (بما فيه خصم المخزون) ولكن **لا يستدعي `AccountingHooks.onOrderCompleted()`** — وهذا **متعمد** لمنع التكرار مع مسار `POST /:id/complete`.  
**الخطر:** إذا استخدم مشغل هذا المسار بدلاً من المسار الرسمي، لن يُنشأ قيد محاسبي (إيرادات + COGS).  
**التوصية:** إضافة تحذير في الـ API Docs أو حجب حالة `completed` من `PUT /status` بالكامل.
